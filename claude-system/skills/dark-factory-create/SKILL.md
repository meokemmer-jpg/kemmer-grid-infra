# /dark-factory-create [CRUX-MK]

## Was ist eine Dark-Factory hier

Cole Medin: Code-Basis die sich selbst pflegt. Wir adaptieren mit Grenzen:

- **Unser Dark-Factory** = Archon-Workflow der **regelmaessig + automatisch** laeuft
- Output wird **ohne Live-Review** in ein **nicht-K_0-relevantes Ziel** geschrieben
- Martin sieht es **asynchron** im Dashboard + Audit-Log
- Bei 2 Fails oder Budget-Break: **Hard-Stop** + Martin-Alert
- **Shadow-Mode 2 Wochen Pflicht** vor Live

## Pflicht-Bedingungen (aus rules/when-to-archon.md Tier 2)

Alle 10 muessen erfuellt sein:

1. Lambda >= 10/Monat
2. Narrow Scope
3. Mind. 80% deterministisch
4. Rollback in < 60s
5. Test-Coverage >= 90%
6. K_0 geschuetzt
7. Q_0 messbar
8. Escalation bei 2 Fails
9. rho-Budget-Hardcap
10. Audit-Trail

Wenn NEIN zu einer: Dark-Factory ablehnen -> Tier 1 Archon-Workflow reicht.

**Stop-Gate:** Decision Card an Martin -> Martin-Approval -> dann weiter.

### Phase 2: SPEC - 3-Layer-Architektur

Layer A (Deterministic Gates): Bash/Python Scripts die vor UND nach jedem AI-Node laufen
- Pre-Node: Input-Validation, CRUX-Check, Budget-Check
- Post-Node: Output-Schema-Validation, Test-Run, Diff-Size-Check

Layer B (AI-Nodes): Claude-Nodes die **keinen** Merge/Push allein machen duerfen
- Jeder AI-Output geht durch Layer-C

Layer C (Escalation): Logic die bei Fails automatisch eskaliert
- 1. Fail -> Retry mit neuer Session
- 2. Fail -> Hard-Stop + BULLETIN-Alert + Martin-Notification (Desktop + Email optional)
- 3. Shadow-Mode -> Martin approved manually die ersten 14 Runs

### Phase 3: IMPLEMENT

#### 3.1 Archon-Workflow bauen
Nutze `archon-workflow-create` mit template `custom.yaml.template`, aber mit zusaetzlichen Nodes:

- `input-validation` (Bash): Prueft Input gegen Schema
- `crux-gate` (Bash): crux-validate.py
- `shadow-check` (Bash): Liest `.archon/dark-factory/<name>/shadow-mode.flag`. Wenn present -> schreibt Output in `shadow/` statt Live-Target.
- `budget-init` (Bash): Budget-Hardcap
- Main-AI-Nodes (Claude, fresh sessions)
- `output-schema-validate` (Bash/Python): JSON-Schema oder Regex-Match
- `test-run` (Bash): pytest / mypy / custom test scripts gegen Output
- `diff-size-check` (Bash): Wenn Diff >> Schwelle -> Fail (keine Massen-Aenderungen)
- `escalation-check` (Python): Liest `.archon/dark-factory/<name>/fail-counter`. Wenn >= 2 -> Stop.
- `audit-log` (Bash): Append in `audit/dark-factory.jsonl`
- `write-target` (Bash): Nur im Live-Mode, sonst Shadow

#### 3.2 Scheduled Task einrichten
Windows Task Scheduler:
- Trigger: cron-equivalent (taeglich / stuendlich / bei Commit)
- Action: `archon workflow run <name> --branch dark/<name>-$(date +%Y%m%d%H%M) "<auto-generated-input>"`
- Budget: separates Tages-Limit in env

#### 3.3 Rollback-Script
```bash
# rollback-<dark-factory-name>.sh
cd <target-repo>
git log --since="1 hour ago" --format=%H --author="archon"
# Revert last N commits by archon
for hash in $(git log --since="1 hour ago" --format=%H --author="archon"); do
  git revert --no-edit $hash
done
git push origin master
```

#### 3.4 Kill-Switch
```bash
# kill-dark-factory-<name>.sh
touch ".archon/dark-factory/<name>/DISABLED"
# Workflow-Erster-Node prueft auf DISABLED-File und exitet
```

### Phase 4: TEST - Shadow-Mode 14 Tage

Touch `.archon/dark-factory/<name>/shadow-mode.flag`.

Alle Runs schreiben nach `shadow/` statt Live. Martin reviewed jeden Run:
- Schreibt approved / rejected in `.archon/dark-factory/<name>/review-log.jsonl`
- Nach 14 Tagen: Wenn Approval-Rate >=95%, remove shadow-mode.flag -> Live.
- Wenn <95%: Extend Shadow-Mode + Workflow verbessern.

### Phase 5: REFINE - Self-Improvement-Loop

**Jeder Fail wird zum Test.** Nach jedem Fail:
1. Auto-generated Test Case in `.archon/dark-factory/<name>/tests/` hinzufuegen
2. Workflow um einen Check-Node erweitern der diesen Fall abfaengt
3. Version-Bump im manifest
4. Knowledge-Diff schreiben (was war der Fail, wie wurde er gefangen)

Siehe Skill `dark-factory-evolve` fuer automatisiertes Strengthening.

## Artifacts-Struktur

```
C:/Users/marti/Projects/learning-archon/.archon/dark-factory/<name>/
├── config.yaml              (Scope + Pflicht-Bedingungen + rho-Schaetzung)
├── workflow.yaml            (Archon-Workflow, Layer A+B+C)
├── schema.json              (Input/Output-JSON-Schema)
├── tests/                   (pytest + custom tests gegen Output)
├── shadow-mode.flag         (existiert = Shadow, nicht existiert = Live)
├── fail-counter             (int, reset bei Success)
├── review-log.jsonl         (Martin-Reviews im Shadow-Mode)
├── scheduled-task.xml       (Windows Task Scheduler export)
├── rollback.sh
├── kill.sh
└── DISABLED                 (Kill-Switch, bei Existenz wird der Workflow sofort nicht ausgefuehrt)
```
